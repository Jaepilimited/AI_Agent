## 6. 개선 권고사항

### 6.1 긴급 (P0) -- Shopify 반품 쿼리 타임아웃

- **문제**: Shopify 반품 관련 쿼리 7건 전수 타임아웃 (122초)
- **원인 추정**: 반품 데이터 테이블의 BigQuery 쿼리 최적화 부족 또는 잘못된 테이블 참조
- **조치 방안**:
  1. Shopify 반품 테이블 스키마 확인 및 SQL 프롬프트에 반품 관련 컬럼 명시
  2. BigQuery 파티셔닝/클러스터링 설정 확인
  3. SQL Agent의 반품 쿼리 생성 로직 점검 (JOIN 조건, WHERE 절 최적화)
  4. API 타임아웃을 150초로 확장하거나, 반품 전용 프리컴파일 쿼리 적용

### 6.2 높음 (P1) -- 스마트스토어 리뷰 테이블 성능

- **문제**: RV-SM 45건 중 21건 실패 (Pass Rate 53.3%), 평균 81.0초
- **원인 추정**: 스마트스토어 리뷰 테이블 데이터량이 크거나 SQL 생성 시 불필요한 전체 스캔 발생
- **조치 방안**:
  1. 스마트스토어 리뷰 테이블의 파티셔닝 (날짜 기준) 적용
  2. SQL 프롬프트에 LIMIT, 날짜 범위 제한 강제 규칙 추가
  3. 자주 사용되는 집계 쿼리를 Materialized View로 사전 생성
  4. 제품별/채널별 집계를 위한 인덱스 최적화

### 6.3 중간 (P2) -- 영문 제품명 매핑

- **문제**: 영문 키워드(Cream, Ampoule)로 검색 시 한국어 제품명과 매칭 실패 (3건)
- **조치 방안**:
  1. SQL Generator 프롬프트에 영한 제품명 매핑 테이블 추가
     - Cream -> 크림, Ampoule -> 앰플, Toner -> 토너, Serum -> 세럼 등
  2. LIKE 검색 시 영문/한국어 양방향 OR 조건 자동 생성 규칙 적용
  3. 제품명 정규화 테이블(product_name_alias) 생성 고려

### 6.4 중간 (P2) -- 플랫폼/아마존 검색 쿼리 최적화

- **문제**: PL 평균 68.2초, AZ 평균 63.0초로 WARN 수준
- **조치 방안**:
  1. 가격 비교, 전환율 계산 등 복합 연산 쿼리의 SQL 패턴 최적화
  2. 채널별 집계 쿼리에 APPROX_COUNT_DISTINCT 등 근사 함수 활용
  3. 자주 조회되는 채널별/국가별 집계를 BigQuery Scheduled Query로 사전 계산

### 6.5 낮음 (P3) -- 쇼피 리뷰 성능 개선

- **문제**: RV-SH 45건 중 11건 FAIL (Pass Rate 75.6%), 평균 49.7초

  1. 쇼피 리뷰 테이블 파티셔닝 확인 (리뷰 날짜 기준)
  2. 날짜별/제품별 GROUP BY 쿼리 최적화
  3. 최근 N개월 데이터 한정 쿼리 기본 적용
